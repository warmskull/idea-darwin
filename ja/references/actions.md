# 標準アクションリファレンス

このファイルは、Idea Darwin エンジンの各標準アクションの目的、適用シナリオ、実行要件を説明します。

## 1. タグ付け

**目的**：スケジューリング、類似性マッチング、交配のためのアイデア分類を標準化する。

**タグの次元**：
- トピックタグ：AI / 教育 / SaaS / コンテンツ / クリエイターエコノミー / ...
- 問題タイプ：効率化 / イノベーション / ビジネス検証 / コラボレーション / 自動化
- ステージタグ：seed / exploring / refining / validated / dormant
- スタイルタグ：高イノベーション / ハイリスク / 実用的 / 研究志向 / プラットフォーム型
- リソースタグ：低コスト / 高コスト / データ依存 / エンジニアリング重視 / ユーザーオペレーション必要

**実行要件**：
- 各アイデアには最低3つ、最大8つのタグが必要
- タグは具体的に — 広すぎるラベルを避ける（例：「テクノロジー」は曖昧すぎる、「自然言語処理」を使用）
- タグはカードの YAML フロントマター `tags` フィールドに記入

---

## 2. カード生成

**目的**：自然言語のアイデアをシステムが処理できる構造化カードに変換する。

**生成内容**：
- コアクエスチョン：このアイデアが本当に答えようとしているものは何か？
- 完全な説明：起源、重要性、現時点で最も有望な形をカバーする2〜4段落
- 現在のテンション：重要な矛盾や未解決の問い
- 深化の方向性
- 交配候補

**実行要件**：
- `assets/card-template.md` のフォーマットに厳密に従う
- コアクエスチョンは明確で適切に構成された問いでなければならない
- 完全な説明は構造化された蒸留であり、元のテキストのコピーペーストではない

---

## 3. 深化

**目的**：曖昧なアイデアを明確にし、粗いアイデアを完全にする。

**使用タイミング**：
- アイデアにポテンシャルがあるが表現が不十分
- ロジックが不完全
- ターゲットユーザー、シナリオ、パスの特定が必要

**深化の角度**：
- 5W1H（誰が / 何を / どこで / いつ / なぜ / どのように）
- 第一原理分解
- ユーザーシナリオの具体化
- 技術ロードマップのマッピング
- ビジネスモデルの探索
- リスクと制約の特定
- MVPパスの設計
- 失敗ポイントの列挙

**実行要件**：
- これは書き直しではない — 論理的なギャップを埋めること
- アイデアの核心的意図を保持する
- 出力には以下を含む：充実した説明、核心的前提、ユーザーシナリオ、MVPバージョン、リスク、推奨次アクション
- カードの「完全な説明」と「さらなる深化の方向性」を更新

---

## 4. 派生

**目的**：アイデアをバリアントに分割し、探索空間を拡大する。

**派生の次元**：
- 異なるユーザーセグメント
- 異なるビジネスモデル
- 異なる技術的実装
- 異なるシナリオ
- 異なる難易度
- 異なるプロダクト形態
- 異なるタイムホライズン（短期 / 長期）

**実行要件**：
- 1回の派生で2〜3の子アイデアを生成（組み合わせ爆発を防ぐため3を超えない）
- 各子は親のコアDNAを保持しつつ、明確な差別化要因を持つ
- 単に言い換えただけの重複バリアントを生成しない
- 各子には新しい IDEA-XXXX IDが自動的に割り当てられる
- 親の `child_ids` と各子の `parent_ids` を更新
- 各子にはカード全体の生成と初期スコアリングが必要

---

## 5. 交配 / ハイブリダイズ

**目的**：異なるアイデアを融合し、新しい視点を生み出す。

**2つのタイプ**：
1. **近接交配**：類似アイデアとの交配 → 収束的最適化
2. **遠隔交配**：無関係なアイデアとの交配 → 革新的ブレイクスルー（主に撹乱ラウンドで使用）

**交配分析フロー**：
1. 2つのアイデアの共通点を分析
2. 補完的な強みを分析
3. 対立点と境界を分析
4. 1〜3の新しいハイブリッドアイデアを生成
5. 各ハイブリッドのコアメカニズムと価値を評価

**実行要件**：
- 新しいアイデアには自動的にIDが割り当てられる
- 新しいアイデアの `parent_ids` には両方のソースアイデアIDを含む
- 両方のソースアイデアの `child_ids` を更新
- 交配は機械的な結合ではない — 各アイデアから価値ある部分を抽出し、真に新しいコンセプトを形成する

---

## 6. 批評

**目的**：弱点を強制的に発見し、システムがエコーチェンバーになるのを防ぐ。

**チェックリスト**：
- 自己矛盾はないか？
- 前提が強すぎないか？
- 実需が欠如していないか？
- コストが法外に高くないか？
- 既存アイデアの再パッケージではないか？
- 検証パスがないか？
- 最も失敗しやすいポイントはどこか？

**実行要件**：
- 出力：コア欠陥リスト、リスクレベル、継続推奨かどうか、最初に修正すべきこと
- 批評後、カードの「現在のテンション」を更新
- 重大な問題が見つかった場合、対応するスコア次元を下げる

---

## 7. 検証

**目的**：アイデアに引き続きリソースを投資する価値があるかを判断する。

**第1層：致命的欠陥チェック**（4つのはい/いいえの質問）：
1. 論理的欠陥があるか？（核心的前提が自己矛盾しているか、明らかに無効か？）
2. 実需が欠如しているか？（特定可能なターゲットユーザーやユースケースがないか？）
3. 完全に実現不可能か？（技術/リソース/パスに克服不能な障壁があるか？）
4. 既存アイデアとほぼ重複しているか？（単に言い換えたバージョンか？）

いずれかが「はい」→ `validated: false`。

**第2層：価値判断**（第1層がすべて「いいえ」の場合のみ）：
1. 引き続き投資する価値はあるか？
2. 次のアクションは何か？
3. 現在の最大リスクは何か？

**実行要件**：
- 検証失敗 → Priority Score を下げ、ユーザーに報告
- ユーザーが休眠か削除かを決定 — システムは推奨するのみ
- 検証パス → 状態の昇格を推奨する場合がある（例：exploring → refining）

---

## 8. 統合 / 重複排除

**目的**：高度に冗長なコンテンツでアイデアプールが肥大化するのを防ぐ。

**トリガー条件**：
- 2つのアイデアが同じ核心的前提を共有
- 表現の違いだけ
- タグが大幅に重複し、説明が類似

**処理方法**：
- メインアイデア + バリアント記録に統合
- 吸収されたアイデアを `merged_into: IDEA-XXXX` としてマーク
- ブランチを保持するが優先度を下げる

**実行要件**：
- 毎ラウンド終了時にプール全体の類似性チェックが必須
- 類似性判断はタグの重複率 + コアクエスチョンの類似度に基づく
- 統合時は、より豊富な情報を持つバージョンをメインアイデアとして保持
- 統合履歴をカードの「変更ログ」に記録
