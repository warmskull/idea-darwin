# 标准动作参考

本文件描述 Idea Darwin 引擎中每种标准动作的用途、适用场景和执行要求。

## 1. 打标签

**用途**：标准化 idea 分类，服务于调度、相似度匹配和交叉繁殖。

**标签维度**：
- 主题标签：AI / 教育 / SaaS / 内容 / 创作者经济 / ...
- 问题类型：效率提升 / 创新突破 / 商业验证 / 协作 / 自动化
- 阶段标签：seed / exploring / refining / validated / dormant
- 风格标签：高创新 / 高风险 / 实用型 / 研究型 / 平台型
- 资源标签：低成本 / 高成本 / 数据依赖 / 工程密集 / 需要运营

**执行要求**：
- 每个 idea 至少 3 个标签，不超过 8 个
- 标签要具体 — 避免过于宽泛（如"技术"太笼统，应使用"自然语言处理"）
- 标签写入卡片 YAML frontmatter 的 `tags` 字段

---

## 2. 生成卡片

**用途**：将自然语言描述的 idea 转化为系统可处理的结构化卡片。

**生成内容**：
- 核心问题：这个 idea 到底在回答什么？
- 完整描述：2–4 段覆盖起源、重要性和当前最有潜力的形态
- 当前张力：关键矛盾或未决问题
- 可深化方向
- 可交叉对象

**执行要求**：
- 严格遵循 `assets/card-template.md` 格式
- 核心问题必须是一个清晰、完整的问题
- 完整描述是结构化提炼，不是原文的复制粘贴

---

## 3. 深化

**用途**：把模糊的想法变清晰，把粗糙的想法变完整。

**适用场景**：
- idea 有潜力但表述不清
- 逻辑链不完整
- 需要明确目标用户、场景或路径

**深化角度**：
- 5W1H（谁 / 什么 / 哪里 / 何时 / 为什么 / 怎么做）
- 第一性原理分解
- 用户场景具象化
- 技术路线图梳理
- 商业模式探索
- 风险与约束识别
- MVP 路径设计
- 失败点枚举

**执行要求**：
- 这不是简单改写 — 而是补齐逻辑缺口
- 保留 idea 的核心意图
- 输出必须包含：充实后的描述、核心假设、用户场景、MVP 版本、风险、建议下一步
- 更新卡片的"完整描述"和"可继续深化的方向"

---

## 4. 派生

**用途**：将一个 idea 分裂为多个变体，扩大搜索空间。

**派生维度**：
- 不同用户群
- 不同商业模式
- 不同技术实现
- 不同场景
- 不同难度级别
- 不同产品形态
- 不同时间跨度（短期 / 长期）

**执行要求**：
- 每次派生产出 2–3 个子 idea（不超过 3 个，防止组合爆炸）
- 每个子 idea 必须保留母体的核心基因，但有明确差异点
- 不要生成只是换了说法的重复方案
- 每个子 idea 自动分配新的 IDEA-XXXX ID
- 更新母体的 `child_ids` 和每个子代的 `parent_ids`
- 每个子 idea 都需要完整的卡片生成和初始评分

---

## 5. 交叉 / 杂交

**用途**：融合不同 idea 以产生新视角。

**两种类型**：
1. **近邻交叉**：与相似 idea 交叉 → 收敛优化
2. **远距交叉**：与不相关 idea 交叉 → 创新突破（主要在扰动轮使用）

**交叉分析流程**：
1. 分析两个 idea 的共同点
2. 分析互补优势
3. 分析冲突和边界
4. 产出 1–3 个新的杂交 idea
5. 评估每个杂交体的核心机制和价值

**执行要求**：
- 新 idea 自动分配 ID
- 新 idea 的 `parent_ids` 包含两个源 idea 的 ID
- 更新两个源 idea 的 `child_ids`
- 交叉不是机械拼接 — 要提取各自的有价值部分形成真正的新概念

---

## 6. 批判

**用途**：强制寻找弱点，防止系统沦为回声室。

**检查清单**：
- 是否自相矛盾？
- 假设是否过强？
- 是否缺乏真实需求？
- 成本是否过高？
- 是否只是换了包装的旧想法？
- 是否没有验证路径？
- 最可能在哪一步失败？

**执行要求**：
- 输出：核心漏洞列表、风险等级、是否建议继续、最应先修什么
- 批判后更新卡片的"当前张力"
- 如发现严重问题，降低对应评分维度

---

## 7. 验证

**用途**：判断一个 idea 是否值得继续投入资源。

**第一层：硬伤检查**（4 个 Yes/No 问题）：
1. 是否存在逻辑硬伤？（核心假设是否自相矛盾或明显不成立？）
2. 是否缺乏真实需求？（能否找到明确的目标用户或使用场景？）
3. 是否完全不可落地？（技术/资源/路径上是否有无法克服的障碍？）
4. 是否与已有 idea 高度重复？（是否只是换了说法的旧 idea？）

任何一项为"Yes" → `validated: false`。

**第二层：价值判断**（仅当第一层全部为"No"时执行）：
1. 是否值得继续投入？
2. 下一步应做什么？
3. 当前最大风险是什么？

**执行要求**：
- 验证失败 → 降低 Priority Score，报告给用户
- 用户决定是否休眠或移除 — 系统仅推荐
- 验证通过 → 可推荐状态晋升（如 exploring → refining）

---

## 8. 合并 / 去重

**用途**：防止 idea 池被高度冗余的内容膨胀。

**触发条件**：
- 两个 idea 共享相同的核心假设
- 仅在措辞上有差异
- 标签高度重叠且描述相似

**处理方式**：
- 合并为主 idea + 变体记录
- 标记被吸收的 idea 为 `merged_into: IDEA-XXXX`
- 保留分支但降低优先级

**执行要求**：
- 每轮结束前必须对全池进行相似度检查
- 相似度判断基于标签重叠率 + 核心问题相似度
- 合并时保留信息更丰富的版本作为主 idea
- 在卡片的"变更历史"中记录合并历史
