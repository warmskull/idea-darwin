# プロンプトテンプレート

各アクションのプロンプトテンプレート。アクション実行時に変数を実際の値に置き換え、思考ガイドとして使用してください。

---

## 共通テンプレート構造

すべてのアクションプロンプトは以下の構造に従います：

```
あなたは「アイデア反復インキュベーションシステム」の {action_name} モジュールです。

あなたのタスクは、以下のアイデアに対して {action_name} 操作を実行することです。

【現在のアイデアカード】
{idea_card}

【現在のラウンドコンテキスト】
- ラウンド: {round_id}
- 現在のステージ: {stage}
- 利用可能なバジェット: {budget_level}
- 関連アイデア: {related_ideas_summary}

【操作目標】
{goal_description}

【操作要件】
1. 論理的厳密さを維持する — 曖昧な穴埋めは不可。
2. アイデアの核心的意図を可能な限り保持する。
3. 明らかな欠陥を発見した場合は指摘する。
4. 新しいアイデアを生成する場合、元のアイデアとの明確な関係が必要。
5. 出力は構造化されていなければならない。
```

## 深化プロンプト

```
このアイデアに対して「深化」操作を実行してください。これは表面的な書き直しではなく、論理構造を埋めることが目的です。

以下の角度から展開してください：
1. 実際に何の問題を解決するのか？
2. ターゲットユーザーは誰か？
3. なぜこのアプローチが有効なのか？
4. 最小実行可能バージョンは何か？
5. 最大のリスクは何か？
6. まだ検証が必要な主要な前提は何か？

出力：
- 充実したアイデアの説明
- 核心的前提
- ユーザーシナリオ
- MVPバージョン
- リスクと未解決の問い
- 推奨次アクション
```

## 派生プロンプト

```
以下のアイデアに基づき、2〜3の高品質なバリアントを派生させてください。

以下の次元に沿って分解してください：
- ユーザーセグメントの違い
- シナリオの違い
- ビジネスモデルの違い
- プロダクト形態の違い
- 技術パスの違い

要件：
1. 各バリアントは元のアイデアのコアDNAを保持すること
2. ただし、明確な差別化要因を持つこと
3. 単に言い換えただけの重複バリアントを生成しないこと

出力フォーマット（各バリアント）：
- バリアントタイトル
- コアの変更点
- 保持する価値がある理由
- 元のアイデアとの関係
```

## 交配プロンプト

```
以下の2つのアイデアを交配させ、新しい統合方向を見つけてください。

【アイデア A】
{idea_a_card}

【アイデア B】
{idea_b_card}

まず分析してください：
1. 共通点は何か？
2. 相互補完するところはどこか？
3. 対立点はあるか？

次に出力してください：
1. 1〜3の新しいハイブリッドアイデア
2. 各ハイブリッドのコアメカニズム
3. この交配が意味を持つ理由
4. 最大のリスク
5. 次のラウンドに進める価値があるか
```

## 批評プロンプト

```
以下のアイデアに対して批評レビューを実行してください。

焦点：
1. 論理的に一貫しているか？
2. 実体のないコンセプトのパッケージングに過ぎないか？
3. 実需が欠如していないか？
4. コストが法外に高くないか？
5. よりシンプルな代替案はないか？
6. このアイデアが最も失敗しやすいポイントはどこか？

出力：
- コア欠陥
- リスクレベル（高 / 中 / 低）
- 推奨：継続するかしないか
- 継続する場合、最初に修正すべきこと
```

## スコアリングプロンプト

```
このアイデアを6つの次元で評価してください（各1〜10）。理由付きで：

- 新規性（重み 0.1）：本当のブレイクスルーがあるか、それとも単なる繰り返しか？
- 実現可能性（重み 0.2）：技術的、リソース的、パス的に達成可能か？
- 価値（重み 0.2）：成功した場合、どれだけのインパクトを生み出せるか？
- 論理性（重み 0.2）：内部的に一貫しており、明らかなギャップがないか？
- 交配ポテンシャル（重み 0.1）：他のアイデアと組み合わせたとき、新しいものを生み出す可能性は？
- 検証可能性（重み 0.2）：実験や最小検証パスを設計できるか？

次に計算：
Survival = 0.10×Novelty + 0.20×Feasibility + 0.20×Value + 0.20×Logic + 0.10×CrossPotential + 0.20×Verifiability

JSON形式で出力：
{
  "novelty": 0,
  "feasibility": 0,
  "value": 0,
  "logic": 0,
  "cross_potential": 0,
  "verifiability": 0,
  "survival_score": 0.0,
  "reasoning": {
    "novelty": "...",
    "feasibility": "...",
    "value": "...",
    "logic": "...",
    "cross_potential": "...",
    "verifiability": "..."
  },
  "suggested_action": "深化 / 派生 / 交配 / 批評 / 休眠"
}
```

## 検証プロンプト

```
このアイデアに対して二層検証チェックを実行してください。

【第1層：致命的欠陥チェック】
以下の4つの質問に回答してください（はい/いいえ + 理由）：
1. 論理的欠陥があるか？（核心的前提が自己矛盾しているか、明らかに無効か？）
2. 実需が欠如しているか？（特定可能なターゲットユーザーやユースケースがないか？）
3. 完全に実現不可能か？（技術/リソース/パスに克服不能な障壁があるか？）
4. 既存アイデアとほぼ重複しているか？（単に言い換えたバージョンか？）

いずれかの回答が「はい」の場合、validated: false とし、簡潔な説明を添える。

【第2層：価値判断】（第1層がすべて「いいえ」の場合のみ）
1. 引き続きリソースを投資する価値はあるか？
2. 次のアクションは何か？
3. 現在の最大リスクは何か？

JSON形式で出力：
{
  "hard_check": {
    "logic_flaw": {"result": "はい/いいえ", "reason": "..."},
    "no_real_demand": {"result": "はい/いいえ", "reason": "..."},
    "not_feasible": {"result": "はい/いいえ", "reason": "..."},
    "duplicate": {"result": "はい/いいえ", "reason": "..."}
  },
  "validated": true/false,
  "value_judgment": {
    "worth_continuing": "はい/いいえ",
    "next_action": "...",
    "biggest_risk": "..."
  }
}
```
